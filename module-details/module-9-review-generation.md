# Module 9: Review Generation System

## 1. Module Overview
The Review Generation System is designed to actively solicit reviews from past customers. It allows businesses to upload bulk customer lists (via CSV) and orchestrates automated SMS or WhatsApp outreach campaigns asking for Google Business Profile reviews. It tracks open rates, click rates, and ultimately links them to actual reviews left on the platform.

## 2. Architecture
- **Frontend Structure**: 
  - `src/app/dashboard/upload/page.jsx`: A drag-and-drop CSV upload interface to ingest large customer lists.
  - `src/app/dashboard/campaigns/page.jsx`: Interface to trigger Review Request campaigns to the ingested customers.
- **Backend Structure**: 
  - `src/app/api/upload/route.ts`: Parses the uploaded CSV file and creates `Customer` records.
  - `src/app/api/campaigns/route.ts`: Manages the creation of outreach campaigns.
- **Service Layer**: 
  - Indirectly uses the SMS/WhatsApp services to send the review request links.
- **Database**: 
  - Uses `Customer`, `Campaign`, and `ReviewRequest` Mongoose models.

## 3. APIs Used
- **Internal APIs**: 
  - `POST /api/upload`: Expects FormData containing a CSV file. Parses it using `csv-parse` and bulk-inserts into MongoDB.
  - `POST /api/campaigns`: Triggers an outreach.
- **External APIs**: 
  - None directly during the upload phase. The Twilio API is utilized when the `Campaign` actively executes.

## 4. MongoDB/Mongoose Structure
- **Collections**:
  - `customers`: Stores `name`, `phone`, `email`, `lastVisit`, `totalSpent`, and `optIn`.
  - `campaigns`: Tracks `name`, `type` (e.g., SMS, WHATSAPP), `status`, `targetAudience`, and `scheduledFor`.
  - `reviewrequests`: The core ledger of outreach. Tracks `customerId`, `channel`, `status` (QUEUED, SENT, CLICKED, REVIEWED), and dates for reminders (`reminder1SentAt`).

## 5. AI Integrations
- While the upload logic itself is not AI-driven, future campaign messages (the actual text sent asking for a review) are planned to be dynamically generated by the Groq API based on the customer's `lastVisit` and `totalSpent` profile.

## 6. Automation & n8n Workflows
- **Triggers**: The system is designed to have n8n poll the `ReviewRequest` collection for `status: 'QUEUED'` records to orchestrate the actual sending of WhatsApp messages with delays and follow-ups. The repository currently lacks the `n8n` `.json` workflows to handle this.

## 7. UI/UX Components
- **Pages**: `/dashboard/upload`, `/dashboard/campaigns`
- **Design System**: Features a visually distinct file-drop zone with animated Lucide icons and an integrated upload progress tracker.

## 8. Environment Variables Needed
None specifically for the upload logic, besides standard database connection strings.

## 9. Execution/Setup Guide
1. Run `npm run dev`.
2. Navigate to `/dashboard/upload`.
3. Create a sample CSV with headers `name,phone,email,lastVisit,totalSpent`.
4. Drag and drop the CSV to ingest the customers into the database.

## 10. Module Dependencies
- **Depends on**: MongoDB for bulk inserts.
- **Depended upon by**: Admin Dashboard Analytics.

## 11. Known Issues / Technical Debt
- **Missing Link Tracking Logic**: The `ReviewRequest` schema has `clicked` and `reviewed` boolean statuses, but there are no dynamic redirect endpoints built (e.g., `GET /r/[unique-id]`) to actually track if a user clicked the SMS link before redirecting them to Google.
- **Deduplication Overwrites**: The `/api/upload` CSV parser uses `$setOnInsert` to prevent duplicate phones, but it does not update existing customer metrics (like `lastVisit`) if the customer already exists in the database.

## 12. Future Scaling Suggestions
- **Smart Link Redirection**: Implement a middleware or edge route (`/go/[id]`) that intercepts the click, sets `ReviewRequest.status = 'CLICKED'`, and redirects to the GMB Review URL.
- **AI Audience Segmentation**: Add a feature to pass the entire ingested `Customer` list to the LLM to automatically generate segmented campaigns (e.g., "Send VIP campaign to customers who spent >$500").
